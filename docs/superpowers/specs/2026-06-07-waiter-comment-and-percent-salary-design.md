# Дизайн: имя официанта в комментарии заказа + новая модель зарплаты (3000/смена + % от продаж)

Дата: 2026-06-07
Клиент: Грузин Кузин (сеть ресторанов на iiko)

## Контекст и цель

Три связанные задачи:

1. **Комментарий заказа.** При создании заказа через iiko Cloud `POST /api/1/order/create`
   мы не можем штатно передать официанта по API. Как обходной путь — пишем имя официанта
   в `comment`: «{ФИО} оформил с приложения», плюс пользовательский комментарий, если он есть.

2. **Хранение персональных процентов официантов.** У каждого официанта свой процент с продаж,
   меняется помесячно. Источник — `ofik_percent_salary.xlsx` («% официантов на май»).

3. **Новый расчёт зарплаты.** За смену 3000 ₸ + персональный процент от продаж смены.
   Зарплата атрибутируется на **дату открытия смены**: смена, открытая 4-го и закрытая 5-го,
   полностью идёт на 4-е число.

## Решения (согласовано с заказчиком)

- % хранится в **отдельной таблице с периодами** (`waiter_sales_percent`).
- База для процента — **продажи внутри окна смены** `[start_time, end_time|now]`.
- 3000 начисляется **за каждую смену** (две смены за день → 6000).
- Комментарий: **имя + пользовательский комментарий** («{ФИО} оформил с приложения. {коммент}»).
- `date_from` импорта = `2026-05-01`; нет записи о проценте на дату → процент = 0.
- В этой функции зарплаты убираем почасовую логику и зависимость от `UserSalary`.
- В `SalaryBreakdown` добавляем `percentAmount` (сумма, начисленная с процента).

## Фича 2 — Хранение персональных процентов (таблица с периодами)

### Модель

`models/waiter_sales_percent.py` → таблица `waiter_sales_percent`:

| поле        | тип            | примечание                          |
|-------------|----------------|-------------------------------------|
| id          | Integer PK     |                                     |
| employee_id | FK employees.id| индекс                              |
| percent     | Numeric(5,2)   | напр. 7.00                          |
| date_from   | Date           | начало действия                     |
| date_to     | Date, nullable | NULL = бессрочно / до замены        |
| created_at  | DateTime       |                                     |
| updated_at  | DateTime       |                                     |

Активный процент на дату X = запись где `date_from <= X AND (date_to IS NULL OR date_to >= X)`.
При нескольких подходящих — берём свежайшую по `date_from`.

### Миграция

`migrations/2026_06_07_add_waiter_sales_percent.sql` + парный `.py` (по образцу существующих
миграций в `migrations/`).

### Скрипт-импортёр

`migrations/import_waiter_percents.py` — запускается вручную, читает `ofik_percent_salary.xlsx`.

- Берём лист `Лист1`, колонки «Имя в системе» и «%». Колонку «гк N» игнорируем (группировка точек).
- `percent` приводим к числу (в файле часть значений строки, часть int).
- **Матчинг имени** (порядок слов и каз. буквы расходятся между xlsx и `employees.name`):
  - нормализация: `lower()`, trim, схлопывание пробелов;
  - свод казахских букв: `қ→к, ә→а, ө→о, ұ→у, ү→у, і→и, ң→н, ғ→г, һ→х`;
  - токенизация по пробелам, сравнение как **множества токенов** (порядок не важен);
  - матч засчитывается при полном совпадении множеств токенов с ровно одним `Employees`
    (фильтр по не удалённым). Неоднозначные (0 или >1 кандидатов) — в отчёт.
- Для сматченных создаём `waiter_sales_percent(employee_id, percent, date_from=2026-05-01, date_to=NULL)`.
  Идемпотентность: перед вставкой проверяем, нет ли уже записи на этот `employee_id` c `date_from=2026-05-01`.
- **Отчёт в stdout:** количество сматченных; список несопоставленных строк xlsx (имя + %)
  и список неоднозначных — для ручного занесения.

## Фича 3 — Новый расчёт зарплаты по дате открытия смены

**Файл:** `services/salary/salary_service.py`, функция `calculate_waiter_salary`.

Заменяем почасовую логику (`UserSalary`/WR1 × часы) на:

```
вход: waiter_id, date X ("DD.MM.YYYY"), organization_id?

employee, user = _resolve_employee_user(...)   # как сейчас

# Смены, ОТКРЫТЫЕ в дату X (атрибуция по дате открытия — ключ фичи 3):
#   start_time::date == X
#   открытая смена (end_time IS NULL) — с защитой 48ч и «не в будущем»,
#   по аналогии с текущим окном открытой смены.
shifts_X = смены employee где date(start_time) == X (+ guard для открытых)

base = 3000 * len(shifts_X)

total_revenue = 0
for s in shifts_X:
    seg_end = s.end_time or now
    revenue_s = Σ DOrder.sum_order  где
        user_id == user.id, deleted == False,
        time_order ∈ [s.start_time, seg_end],
        (organization_id фильтр если задан)
    total_revenue += revenue_s

percent = активный % официанта на дату X (waiter_sales_percent) или 0
percent_amount = round(total_revenue * percent / 100, 2)

salary = base + percent_amount
tables_completed = число заказов, попавших в окна смен
quest_* = как сейчас (не меняется)
total_earnings = salary + quest_bonus
```

### Заполнение `SalaryResponse`

- `salary` = `base + percent_amount`
- `salaryPercentage` = ставка процента (напр. 7.0)
- `totalRevenue` = `total_revenue`, `tablesCompleted` = число заказов в окнах смен
- `bonuses` = 0, `penalties` = 0 (как сейчас), `questBonus`/`questDescription`/`quests` — без изменений
- `totalEarnings` = `salary + quest_bonus`
- `breakdown.baseSalary` = `base` (3000 × N)
- `breakdown.percentage` = ставка процента
- `breakdown.percentAmount` = `percent_amount` (**новое поле**)
- `breakdown.bonuses` / `penalties` / `questRewards` — как сейчас

### Изменение схемы

`schemas/salary.py` → в `SalaryBreakdown` добавить `percentAmount: float`.

### Удаляем

- Импорт и использование `UserSalary` в `calculate_waiter_salary`.
- Почасовой расчёт `base_salary = ставка × hours_worked` и саму переменную `hours_worked`
  (для расчёта больше не нужна).

Функция `get_waiter_daily_sales` не меняется.

## Влияние / границы

- Kill switch iiko (`IIKO_REQUESTS_DISABLED`) не затрагивается: фича 1 — это локальная сборка
  строки комментария, реальный трафик уйдёт в iiko только при снятии флага.
- Расчёт зарплаты (фичи 2–3) — чисто БД, без обращений к iiko.
- Существующие данные `UserSalary` остаются в БД, просто перестают использоваться этой функцией.

- **Фича 1:** unit-тест на сборку комментария (есть имя + есть/нет коммент юзера; официант не найден).
- **Фича 2:** тест нормализации/матчинга имён (разный порядок слов, каз. буквы);
  тест выбора активного процента на дату (периоды, NULL date_to, отсутствие записи → 0).
