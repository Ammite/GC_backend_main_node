# Task 9 — Передизайн цикла оформления заказа (iiko Cloud)

**Статус:** to discuss + research (2026-05-23 → ...)
**Заведено:** 2026-05-23

## Контекст

Продолжение [[made_task_8_waiter_order_full_cycle]]. После ответа поддержки iiko
([#ID-349227607]) стало ясно, что часть нашей текущей реализации цикла
**использует неверные endpoint'ы** под наш сценарий.

iiko различает **два сценария** работы:

| # | Кто создал заказ | Endpoint оплаты | Что для него нужно |
|---|---|---|---|
| 1 | Наш бэк (`/order/create`) | **`/order/add_payment`** | поллинг `/commands/status`, External Menu |
| 2 | Касса iikoFront (открыл официант) | `/order/change_payments` | `/order/init_by_table`, `/order/by_table` |

Сейчас у нас в коде: **создаём через `/order/create`** (сценарий 1), но **оплачиваем через `/order/change_payments`** (метод для сценария 2). Это смесь, и она, скорее всего, поэтому и «не сходится».

## КЛЮЧЕВАЯ ПРОБЛЕМА (тейк @baa21v 2026-05-23)

> Когда мы заказ создаём через `/order/create` — даже если передаём `waiterId`, **iiko атрибутирует заказ Интегратору** (нашему API-пользователю). Из-за этого:
> - В отчётах iiko заказы выглядят как «Интегратор открыл/обслуживал», а не реальный официант
> - **Зарплата в iiko рассчитывается неправильно** (% от выручки официанта не начисляется, бонусы за продажи теряются)
> - Менеджеры в iiko-отчётах не видят, кто реально работал смену

**Это критично**, потому что обходного пути на стороне Cloud API мы пока не знаем:
- `/order/create` принимает `waiterId`, но похоже **используется как «обслуживающий», а не «открывший/создатель»**
- Метода «сменить создателя» в Cloud нет
- Возможно, в реальности iiko атрибутирует именно по «создателю»

**Из этого следует**: даже если мы починим оплату (`change_payments` → `add_payment`) и поллинг — мы НЕ решим проблему атрибуции зарплаты. А это именно то, ради чего бизнес и нанимает интеграцию.

## Возможные пути решения проблемы атрибуции

### Путь A — выяснить, что именно использует iiko для атрибуции
Открытый вопрос: какое поле iiko OLAP-отчёт «Зарплата» использует для определения «чей заказ»? Если `waiterId` — то наша текущая реализация **правильна**, и проблема не в API, а в том, как клиент смотрит iikoFront UI (где «Открыл» — это всегда Интегратор, но это не используется в расчёте).

**Проверка:** на восстановленной учётке дёрнуть OLAP-отчёт `reportType="SALES"` или `"EMPLOYEES"` с группировкой по официанту за период тестового заказа. Если в отчёте видны имена реальных официантов — всё ок. Если только «Интегратор» — проблема подтверждена.

### Путь B — переход на сценарий 2 (заказы создаются на iikoFront)
Если путь A показал, что атрибуция действительно ломается через `/order/create`:
- Официант **обязан** открыть стол на iikoFront-кассе
- Наш бэк подхватывает заказ через `/order/init_by_table` + `/order/by_table`
- Дальше дополняет (`add_items`), оплачивает (`change_payments`), закрывает (`close`)

**Минус:** официанту нужно держать iikoFront-кассу открытой. Пропадает идея «полностью в нашем приложении».

### Путь C — гибрид
- Мелкие/быстрые сценарии (доставка, take-away) — сценарий 1 (через API)
- Зал/обслуживание — сценарий 2 (через кассу)
- Логика разделения по `tableId IS NULL` или по флагу

### Путь D — найти и использовать недокументированный метод
- iiko Cloud мог не показывать в публичной доке метод «сменить waiter/operator»
- Поспрашивать в поддержке iiko уточнение: «есть ли метод установить waiter как создателя заказа?»

## Подзадачи (отдельные, могут идти параллельно)

### 9.1 — Поллинг `/api/1/commands/status` (КРИТИЧНО, делаем первым)
Все методы iiko Cloud — асинхронные команды, возвращают `correlationId`.
Сейчас мы получаем 200 OK + correlationId и считаем «успех», хотя реальная команда могла упасть через секунду.

Что нужно:
- Хелпер `IikoService.wait_command(correlation_id, timeout=10s, interval=0.5s)`
- Вызывать после всех команд: `create`, `add_items`, `add_payment`/`change_payments`, `close`, `cancel`
- Возвращать `state="Success"` или поднимать понятную ошибку с описанием от iiko

Это **прерывное изменение**: после внедрения мы начнём видеть реальные ошибки, которые раньше глотались. Возможно, обнаружим, что часть текущих сценариев на самом деле **не работают** в iiko, просто мы об этом не знали.

### 9.2 — `add_payment` вместо `change_payments` для сценария 1
Сейчас `change_payments_in_iiko` шлёт `POST /api/1/order/change_payments`. Это для сценария 2.
Для нашего сценария 1 (`/order/create`) iiko рекомендует `POST /api/1/order/add_payment`.

Что нужно:
- Метод `IikoService.add_payment_to_order(order_id, payment)`
- В `pay_order_endpoint` развилка: если заказ создан через `/order/create` → `add_payment`; если через `init_by_table` → `change_payments`
- Отметить в `DOrder.external_data.iiko_creation_method` («api_create» vs «init_by_table»)

Перед этим перепроверить документацию: какие поля payload отличаются между двумя методами.

### 9.3 — Init_by_table + фильтрация заказов по официанту (обсуждали 2026-05-23)
Метод `IikoService.get_cloud_orders_by_table` определён, но не вызывается.
Добавить `init_by_table` (его нет совсем).

**Главная идея @baa21v:** после `init_by_table`/`by_table` мы можем смапить iiko-officianta в нашего User и сохранить в `DOrder.user_id` — тогда **существующий** `GET /orders?user_id=X` без изменений вернёт все заказы официанта, и подобранные с кассы, и созданные у нас.

**Архитектура:**
- `DOrder.user_id` сейчас = «кто создал в нашем приложении». При sync из iiko будем писать туда `User.id` официанта (резолв через `Employees.iiko_id == waiterIikoId` → `User.iiko_id == Employees.iiko_id`).
- Существующий фильтр `user_id` в `GET /orders` работает без правок.
- В `external_data`: `source="init_by_table"`, `iiko_raw=<full response>`, `waiterIikoId`, `waiterId`.
- Заказы, у которых iiko-waiter не нашёлся в нашей БД — `user_id=None` (бесхозные), для них отдельный фильтр в UI «неназначенные» (для менеджера).

**Sync-сервис:**
```python
async def sync_active_orders_from_iiko(db, organization_id):
    org = db.query(Organization).get(organization_id)
    tables = db.query(Table).join(...).all()
    for table in tables:
        iiko_orders = await iiko_service.get_cloud_orders_by_table(
            organization_id=org.iiko_id_cloud, table_id=table.iiko_id)
        for iiko_order in iiko_orders or []:
            upsert_iiko_order_to_dorder(db, iiko_order, table, org)
```

**Upsert:** найти DOrder по `iiko_id`, если есть — обновить `sum_order/items/state` (НЕ перезатирать `user_id` при наличии); если нет — создать с `user_id = waiter_user.id` (или None).

### 9.5 — Назначение официанта (обсуждали 2026-05-23)
Сценарий: заказ открыт на кассе iikoFront → мы его подбираем через `init_by_table` → в нашем приложении кто-то его обслуживает. Кто?

Обсуждённые варианты (без принятого решения):

**Вариант A — «iiko-truth»:** доверяем waiterId из iiko, ничего не переназначаем. Менять — только на кассе. Прозрачно для зарплаты, но негибко.

**Вариант B — «локальный override»:** `external_data.iiko_waiter_id` (для iiko) + `assigned_user_id` (для нашего UI). UI и зарплата по нашим данным. Расходимся с iiko-отчётами.

**Вариант C — гибрид:** если у iiko-waiter есть user в нашей БД → автоматом. Если нет / нет вообще → заказ «бесхозный», любой может «подобрать» через `POST /orders/{id}/claim`. Менеджер видит список бесхозных.

Решение зависит от ответа клиента на вопросы:
1. Когда официант открывает стол на iikoFront — он сам себя назначает или «default cashier»?
2. Если у официанта смена сорвалась — менеджер в нашем приложении должен уметь перевешать заказ?
