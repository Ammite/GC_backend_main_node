# Канбан задач — Грузин Кузин

Проект: приложение для сети ресторанов "Грузин Кузин" на базе бэкенда `backend_main_node` (интеграция с iiko).

## Как устроен канбан

В этой папке хранятся файлы задач. Каждая задача — отдельный `.md` файл.

### Формат имени файла

- **Активная задача:** `task_<номер>_<краткое_имя>.md`
  - Пример: `task_1_create_menu_endpoint.md`
- **Выполненная задача:** перед `task` ставится `made_`
  - Пример: `made_task_1_create_menu_endpoint.md`
  - То есть когда задачу закрыли — просто переименовываем файл, добавляя префикс `made_`.

Номера задач сквозные и не меняются после выполнения.

### Структура файла задачи

Внутри каждого файла задачи рекомендуется держать:

1. **Заголовок** — краткое описание задачи.
2. **Контекст / зачем** — почему делаем, что это даёт.
3. **Что сделать** — список конкретных шагов или требований.
4. **Связанные файлы / эндпоинты** — куда смотреть в коде.
5. **Заметки по реализации** — что важно учесть.
6. **Статус** — `todo` / `in progress` / `done` (плюс дата выполнения, когда закрыли).

## Как смотреть состояние канбана

- Все файлы без префикса `made_` — то, что ещё в работе или ждёт.
- Все файлы с префиксом `made_` — закрытые задачи (история).

Сортировка по номеру = порядок появления задач.

## Инструкция для Claude

Когда пользователь добавляет новую задачу:
- Создаём новый файл `task_<N>_<имя>.md`, где N — следующий по порядку номер.
- Заполняем структуру выше из того, что описал пользователь.

Когда задача выполнена:
- Переименовываем файл, добавляя префикс `made_` (через `git mv` или `mv`).
- В самом файле выставляем `Статус: done` и дату.
