## Контекст
- В какой-то момент пароли юзеров **выгружали** наружу (для раздачи сотрудникам).
- Где именно — не помним: возможно через `POST /employees/recreate-credentials` (есть в коде, возвращает plaintext в ответе) или через какой-то экспорт в Excel/Telegram.
- Беспокойство: возможно, выгрузка прошла **неправильно** — пользователи получили пароли, которые в БД не соответствуют, или пароли соответствуют, но привязаны не к тем логинам.

## Цель
Найти источник выгрузки + сверить с актуальным состоянием БД.

## Где искать
1. **Логи бэка** — `grep -i "recreate\|credentials\|password" logs/` за последние 2-3 месяца. К сожалению, app-логов на диск нет (см. task 2 / made_task_1), только sync_cron.log.
2. **Свежие коммиты** — `git log --all -- 'export*' '*credentials*' '*password*'` — может, кто-то закоммитил скрипт-экспортёр.
3. **Excel-файлы / Telegram** — проверить, не лежат ли в `_kanban/` / `/tmp/` / на чьём-то локале экспорт-файлы. Спросить менеджера: кому/как раздавали?
4. **Эндпоинты, которые могли отдать plaintext:**
   - `POST /employees/recreate-credentials` — возвращает `{"users_created":[{"employee_id":1, "name":"...","login":"...","plain_password":"..."}]}` (если такая функция работает).
   - `POST /employees/create-users` — аналогично может возвращать.
   - Возможно, скрипт `export_*.py` (есть несколько в корне `/srv/project/backend_main_node/`).

## План проверки (когда найдём источник выгрузки)
1. **Получить файл/экспорт** — список (login → plain_password).
2. **Проверить корректность:**
   ```python
   import hashlib
   for login, plain_pwd in exported_list:
       user = db.query(User).filter(User.login == login).first()
       if not user:
           print(f"  ОТСУТСТВУЕТ В БД: {login}")
           continue
       expected_hash = hashlib.sha256(plain_pwd.encode()).hexdigest()
       if expected_hash != user.password:
           print(f"  НЕСОВПАДЕНИЕ: {login} (выданный пароль не работает)")
       else:
           print(f"  OK: {login}")
   ```
3. **Если несовпадения** — массово прогенерить новые через `POST /employees/recreate-credentials` и заново раздать. Старые выгрузки — отозвать.

## Текущее состояние БД (sanity, 2026-05-24)
- Всего юзеров: 256
- Пароль NULL/пустой: 0
- Длина пароля != 64 символов (sha256 hex): 0
- Не-hex символы в пароле: 0
- Дубликатов хешей: 0
- Совпадение с известными слабыми (admin/12345/password/etc): 0

То есть **в БД пароли валидно хешированы и уникальны**. Если выгрузка была — она соответствует тому, что в БД, ровно при условии что её делали через стандартный `recreate-credentials` (он генерирует plain → хеширует → пишет хеш + возвращает plain).

## Сценарии, при которых выгрузка может быть «неправильная»
1. **Кто-то перевыпустил пароли после выгрузки** — старые plaintexts теперь не работают. Решение: сверить (см. выше).
2. **Кто-то добавил юзеров вручную** (через `/register` — он публичный!) и они получили пустой/левый пароль. Решение: посмотреть юзеров без `iiko_id` или с iiko_id, который не в employees.
3. **Выгрузка шла из БД ДО recreate** — пользователи получили старые пароли, в БД новые. Решение: сверить.
4. **Юзера переименовали** (logging changed) — выгрузка имеет старый login. Решение: матчить по iiko_id, не по login.
