## ✅ Что подогнал на бэке под фронт (2026-05-21, после аудита)

Эти изменения **уже в коде**, фронт не трогали:

1. **`POST /register` возвращён** на бэк под совместимость — фронт активно его дёргает (экран `app/auth/registration.tsx`). Поведение как раньше: создаёт User без `iiko_id`/`roles_id`/`app_role`. В docstring помечен как «временный, на решение S1».

2. **`waiter_id` теперь толерантный** в:
   - `GET /waiter/{waiter_id}/salary` — пробуем `Employee.id`, fallback на `User.id → iiko_id → Employee`. Под текущий фронт работает.
   - `GET /waiter/{waiter_id}/quests` — fallback уже был, ничего не менял.

3. **`POST /orders.waiterId`** — новый helper `_resolve_waiter` принимает три формата по очереди: `User.id`, `Employee.id`, `Employee.iiko_id` (числовая строка). Магическое `322256` из фронта теперь сработает корректно. Применено и в `create_order_from_app`, и в `update_order`.

Файлы изменены: `routers/auth/auth.py`, `main.py`, `services/salary/salary_service.py`, `services/orders/orders_services.py`. Smoke-импорт зелёный.

## 🟠 Несовпадения, которые нельзя подогнать на бэке (нужно бизнес-решение)

### S1. Публичная регистрация (`POST /register`) ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** пусть будет открыт. Оставляем как есть.

### S3. Магическое `waiterId=322256` для тестового аккаунта ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** это правильно, костыль для удобства работы. Оставляем как есть, бэк через `_resolve_waiter` его понимает.

### S4. `POST /orders/{id}/cancel` без `removalTypeId`
Фронт шлёт только `{reason: <ru-строка>}`. Бэк подставляет `IIKO_DEFAULT_REMOVAL_TYPE_ID`. Работает, но у iiko есть много типов отмены (брак, ошибка официанта, отказ гостя, и т.п.).

**Вопросы:**
- Нужно ли официанту выбирать тип отмены (dropdown), или дефолтный «системный» достаточно?
- Если выбирать — какие типы доступны и какой UX?

### S5. Расходные накладные / списания без `storeId`/`conceptionId`/`organization_id`
Фронт шлёт только items + дату. В `TESTING_MODE=true` бэк подставляет хардкоды, и они привязаны к одной точке («ИП Шаяхметов»). На других точках iiko вернёт `storeNotFound`.

**Вопросы:**
- Это вообще критичный сценарий (как часто менеджеры списывают со своих точек, кроме «ИП Шаяхметов»)?
- Если да — нужны **per-org дефолты на бэке** (поле в `organizations` для `default_store_iiko_id`, `default_conception_iiko_id`). Фронт остаётся как есть, шлёт `organization_id` (или мы определяем из контекста юзера).

### S6. Предупреждение «не уехало в iiko» при редактировании заказа ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** возвращать 409 Conflict. Жёсткая блокировка — если iiko не поддерживает операцию (удаление позиции, смена количества отправленного заказа), бэк отдаёт 409, фронт показывает «нельзя».

### S7. `GET /payment-types` без `organization_id`
Если в фронте не выбрана точка, в запрос организация не идёт. Бэк отдаёт виды для всех точек, кассир видит лишнее.

**Вопросы:**
- Это реальный сценарий («не выбрана точка»)? Или сейчас фронт всегда выбирает точку перед оплатой?
- Если бывает — на бэке делаем дефолт по `current_user.preferred_organization_id`?

### S8. `GET /menu` без `organization_id`
Аналогично S7, фронт всегда шлёт `{limit: 0}` без `organization_id`. Если меню различается по точкам — официант видит «всё подряд».

**Вопросы:**
- Меню в этой сети одинаковое везде, или различается? Если различается — нужен фильтр.

### S9. Регистрация юзеров для всех сотрудников
`POST /employees/create-users` есть, фронт не вызывает. Если задача — кнопка «создать всем пользователей» — её **сейчас нет в UI**.

**Вопросы:**
- Это нужно как админская кнопка? Где она должна быть (раздел «Сотрудники» в CEO/Manager)?

### S10. Профиль (`GET /profile`, `GET /me`) не используется
Фронт берёт данные профиля из mock + AsyncStorage. Соответственно, `employee_id` в `/profile` пока никем не читается, role-based UI работает на старых данных из логина.

**Вопросы:**
- Сейчас это работает, но это технический долг — фронту надо начать дёргать `/profile` (особенно если меняем модель ролей).
- Когда подключим — отдельной задачей.

## 🟢 Бизнес-вопросы

### 1. Маппинг ролей сотрудник → app_role
Сейчас у нас 30+ кодов ролей iiko. Я сделал heuristic:
- `ADM`, `MN0`, `MN1`, `TekhD`, `GBuh` → **«Менеджер»**
- `WR1` → **«Официант»**
- всё остальное (`PoV`, `BR1`, `Sous chef`, `Lepka`, `Calc`, `buh`, `BChef`, `Mark`, `SG1`, `Rec`, ...) → **«Сотрудник»**

**Вопросы:**
- Кого мы хотим видеть как «Менеджера» / «Владельца»? (Бренд-шеф, тех. директор, главбух — менеджеры или нет?)
- «Сотрудник» — это правильная категория для повара и бармена? Или нужен отдельный «Кухня» / «Бар»?
- Есть ли роль «Владелец», и в каких записях?

### 2. Глубина правок задним числом в iiko ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** 60 дней достаточно. Окно > 60 дней не нужно — даже если редкие правки уйдут «глубже», бизнес готов это принять.

Политика: чанки по дню, максимум окно 60 дней. Закреплено в коде (`get_transactions`, `get_sales`, `get_transactions_by_modification_date`).

### 3. Расходные накладные — привязка к одной точке ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** хардкоды надо снять. Реализовать per-org дефолты — подзадача для отдельной итерации (task_12).

### 4. «Мусорные» виды оплаты с NULL-scope ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** надо почистить. Сделаем после восстановления NEW Server API и пересинка payment_types — пройдём глазами `SELECT * FROM payment_types WHERE organization_iiko_ids IS NULL AND is_deleted=false`.

### 5. Зарплата официанта — формула
До фикса: хардкод 5% от выручки. Сейчас: если есть запись в `user_salaries` — используем `salary` оттуда (то, что iiko синкает); иначе fallback 5%.

**Вопросы:**
- Поле `salary` из iiko — это ставка **за период** (месяц)? За день? Часовая? Бизнес-семантика неясна, и тогда формула «реальный оклад как база зарплаты за день» может быть неправильной.
- У вас вообще официантам платят оклад + процент, только процент, или как?
- Как должны учитываться квесты / штрафы / бонусы?

### 6. Уволенные сотрудники ✅ ЗАКРЫТО (2026-05-24)
**Решение @baa21v:** оставляем как есть. Логин заблокирован (бэк отказывает при `Employee.deleted=true`), User-записи в БД остаются для истории.

### 7. Цикл оформления заказа — функционал
Сейчас в API НЕТ: модификаторов (соусы, размер), split-cheque, prebill, merge-table, move-table, открытия стола с проверкой «уже занят». Tips сохраняются локально, но не уходят в iiko.

**Вопросы:**
- Какие из этих фич вообще нужны для MVP?
- Tips — пробрасывать в iiko как отдельный payment item, или хранить только у нас (текущее поведение)?
- Открытие стола с проверкой «уже есть заказ» — критично или необязательно?

### 8. Удаление позиции / изменение количества у отправленного заказа
iiko Cloud это не умеет. Сейчас бэк **разрешает локально**, но фронту в `message` возвращает «изменение осталось только у нас».

**Вопросы:**
- Этого достаточно (фронт показывает предупреждение пользователю), или хотим **запрещать** с 409 Conflict?
- Если запрещать — нужно UX «можно только добавить» с серой кнопкой «удалить».

### 9. Поле `app_role` — стоит ли расширять?
Сейчас 3 значения: «Владелец», «Менеджер», «Официант». После фикса добавил «Сотрудник» как дефолт для прочих.

**Вопросы:**
- Какие разделы интерфейса доступны каждой роли?
- Нужно ли больше категорий («Кухня», «Бар», «Бухгалтерия»)?

### 10. Раздел «Мотивация» — статусы и фильтры
Я добавил `date_from`/`date_to` с дефолтом «последние 7 дней».

**Вопросы:**
- В UI должны быть **отдельные фильтры по статусу** (активные / истёкшие / выполненные)?
- В табе «История квестов» — какие даты по умолчанию (7 дней / 30 / месяц)?
