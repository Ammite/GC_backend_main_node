Task 6 · Нагрузка iiko / 91-дневный запрос сделано

Корневая причина 4 мая. После фикса лимит iiko 65 дней не нарушается ни одним вызовом, который мы знаем.

Что сделано

  • В get_transactions_by_modification_date (services/iiko/iiko_service.py:1000) окно DateTime.DateTyped сжато с 90 → 60 дней. Бизнес-решение: правки задним числом обычно укладываются в 60 дней.
  • get_transactions и get_sales (iiko_service.py:971-1042) теперь чанкуются по дням — раньше шли одним POST-ом на весь период.
  • POST /sync/by-modification-date (routers/iiko/sync.py:688): дефолт from_date = now-60 вместо 90, валидация to-from ≤ 60 с 400.
  • GET /documents/payrolls (routers/documents/documents.py:774): тоже валидация ≤ 60 дней.

Отложено

  • Общий декоратор чанкования + guardrail в _make_request — на будущее, чтобы новые методы автоматически попадали под защиту.
  • Не-OLAP методы с date-params (get_server_shifts, get_payrolls, get_server_product_expense_report) — пока без чанкования (не самые «жирные»).

Task 3 · Редактирование заказа частично

Фронту теперь видно, что часть изменений не уехала в iiko. Полноценный state machine — отдельной итерацией.

Что сделано

  • PUT /orders/{id}: добавлен diff-анализ. Если у заказа есть iiko_id и фронт меняет что-то, что iiko Cloud не поддерживает (удалить позицию, изменить количество существующей, сменить tableId/waiterId/guests/comment), бэк:
    • пишет в логи warning с конкретным списком расхождений;
    • возвращает message с честным предупреждением «часть изменений осталась только у нас: …».
  • Добавление новых позиций (как и раньше) уходит в iiko через add_items.

Отложено (нужен redesign)

  • State machine DRAFT → SENT → PAID → CANCELLED с enum-полем — сейчас state_order остаётся String.
  • Разделение PUT /orders/{id} на POST/DELETE /orders/{id}/items + PATCH /orders/{id} meta — ломает текущий фронт.
  • 409 Conflict вместо warning при попытке удалить позицию у отправленного заказа — оставлено 200 + предупреждение, чтобы не сломать фронт.
